## 一、项目内容

**仓库:** [LuoVVVV/imtoken-portrait](https://github.com/LuoVVVV/imtoken-portrait)
**线上:** https://imtoken-portrait.tg-codex-210035.workers.dev

> 从以太坊创世到 DeFi 盛世,十年如一日的链上原住民
> 标签:#链上元老 #交易狂魔

## 四、关键踩坑(按踩到的时序)

### 1. CF Pages → Workers 的迁移已发生

**坑:** 一开始按 "Cloudflare Pages + `@cloudflare/next-on-pages`" 的老教程配,build 命令填的是 `npx @cloudflare/next-on-pages`。

**真相:** 2026 年 `@cloudflare/next-on-pages` 已 deprecated。CF 现在 Next.js 预设默认走 **OpenNext 适配器 + Workers**(不是 Pages)。Framework preset 选 Next.js 后,build/deploy 命令默认是 `npm run build` + `npx wrangler deploy`,wrangler 会自动检测 OpenNext。

**修复:** 用 `@opennextjs/cloudflare` 替代,部署到 Workers 而非 Pages。

---

### 2. OpenNext 不支持 Edge Runtime

**坑:**
```
app/api/portrait/route cannot use the edge runtime.
OpenNext requires edge runtime function to be defined in a separate function.
```

**修复:** 删掉 API route 里的 `export const runtime = "edge";`。改用默认 Node 运行时,在 `wrangler.jsonc` 配 `compatibility_flags: ["nodejs_compat"]`。

---

### 3. Next 15.5.2 有 CVE

**坑:**
```
npm warn deprecated next@15.5.2: This version has a security vulnerability. CVE-2025-66478
```

而且 `@opennextjs/cloudflare@1.19.11` 的 peer dep 要求 `next >= 15.5.18`,15.5.2 不满足。

**修复:** `"next": "^15.5.18"`。

---

### 4. `@opennextjs/cloudflare` 不声明 esbuild 但需要它

**坑:**
```
Error [ERR_MODULE_NOT_FOUND]: Cannot find package 'esbuild'
imported from .../@opennextjs/cloudflare/dist/cli/build/bundle-server.js
```

OpenNext 内部用 esbuild 打包 worker,但没把 esbuild 声明为 dependency。

**修复:** 自己 `npm i -D esbuild`。

---

### 5. `npm run build` 必须产出 OpenNext bundle

**坑:** CF 流程是先 `npm run build` 再 `npx wrangler deploy`。如果 `build` script 只是 `next build`,wrangler 会调 `opennextjs-cloudflare deploy`,找不到 `.open-next/` 报:

```
ERROR Could not find compiled Open Next config, did you run the build command?
```

**坑中坑:** 直接把 `build` 改成 `opennextjs-cloudflare build` 会**递归** — OpenNext 内部又调 `npm run build` 来跑 next build,死循环。

**修复:**
```json
"build": "next build && opennextjs-cloudflare build --skipNextBuild"
```
跑两段:先 next build,再 OpenNext bundle(跳过它自己的 next build)。

---

### 6. Next standalone 模式在多级路径下误判 monorepo

**坑:** 项目路径含 `hermes_data` 等多层目录时,`next build` 的 standalone 模式自动检测的"monorepo root"会出错,把输出放到 `.next/standalone/hermes_data/...` 而不是 `.next/standalone/.next/server/...`,OpenNext 找不到 `pages-manifest.json`。

**修复:** 在 `next.config.js` 里钉死路径根:
```js
const nextConfig = {
  output: "standalone",
  outputFileTracingRoot: __dirname,
};
```

CF Linux 环境一般不会触发,但 Windows + 含特殊路径片段时必踩。

---

### 7. CF Build Token 绑定"离职用户"

**坑:** CF Worker 在共享账户 / 子账户里,初始构建 token 可能绑定一个已经离开组织的用户,任何重试都报:

```
Your build is configured with a build token that belongs to a user who has left your organization.
```

**修复:** Worker → Settings → Builds → API token → 创建一个属于自己的新 token。

---

### 8. CF Build 变量 ≠ Worker Runtime 变量

**坑:** 在 Worker 的 "Settings → Build" 区下的 "Variables and secrets" 里加 API key,运行时仍然 `process.env.XXX === undefined`,API 报 500 "服务端缺少 API key 配置"。

**真相:** 那一块只是 build 步骤可用的环境变量。Worker 运行时变量是 **另一处** Settings 的 "Variables and secrets" 区(顶级,不在 Build 子菜单里)。两套要分别配。

**修复:** 同样的 5 个变量(`ETHERSCAN_API_KEY` / `MIMO_API_KEY` / `MIMO_BASE_URL` / `LLM_MODEL` / `NODE_VERSION`)既加在 Build 区(NODE_VERSION 真的需要)也加在 Worker runtime 区。

---

### 9. `ANTHROPIC_API_KEY` 系统变量会被 Claude Code 等工具劫持

**坑:** 本地开发时,如果用 `ANTHROPIC_API_KEY` 命名 LLM key,Claude Code 等 CLI 工具的系统级 `ANTHROPIC_*` 环境变量会把它覆盖,把请求劫持到本地代理上。

**修复:** 用专用名 `MIMO_API_KEY` / `MIMO_BASE_URL`,代码里 fallback 到 `ANTHROPIC_*` 作为兜底:
```ts
const anthropicKey = process.env.MIMO_API_KEY ?? process.env.ANTHROPIC_API_KEY;
const anthropicBaseUrl = process.env.MIMO_BASE_URL ?? process.env.ANTHROPIC_BASE_URL;
```

---

### 10. MiMo Pro 系列默认开 thinking,会把 max_tokens 吃光

**坑:** 用 SDK 直调 `messages.create({ ... })` 时,响应里 content 为空,stop_reason 异常。MiMo Pro 默认 thinking 是开的,会吃光预算。

**修复:** 不走 SDK,自己 fetch 调 `/v1/messages`,把 `thinking: { type: "disabled" }` 显式塞进 body:
```ts
const requestBody = {
  model: "mimo-v2.5-pro",
  max_tokens: 1500,
  thinking: { type: "disabled" },
  messages: [{ role: "user", content: prompt }],
};
```

---

### 11. Anthropic 的 Opus 4.7 安全分类器挂掉时,Claude Code auto mode 全瘫

**坑:** 在 Claude Code 的 auto mode 下,bash 写操作每次都要过 Anthropic 服务端的 Opus 4.7 分类器审核。这模型挂了,所有写操作 100% 被 block,连 `git add` 都跑不了。

**修复(多层防御):**

1. 在 `~/.claude/settings.json` 加 `permissions.allow` 白名单 — npm / git / gh / docker / 网络诊断等都放行
2. 把 `defaultMode` 从 `"auto"` 改成 `"acceptEdits"` 可绕开 classifier 依赖
3. 工作流末尾保留"实在不行让用户手动跑"的 fallback 心理准备

## 五、可复用的部分

### A. Cloudflare Workers + Next.js 部署模板

整套 `wrangler.jsonc` + `open-next.config.ts` + `next.config.js` + `package.json` scripts 现在是经过实战的,任何 Next 15 项目可以直接搬。要点:
- `nodejs_compat` flag
- `outputFileTracingRoot: __dirname`
- `next build && opennextjs-cloudflare build --skipNextBuild`
- 必须把 `esbuild` 放 devDeps

### B. Etherscan 派生标签算法

`lib/etherscan.ts` 里基于:
- 链上年龄
- DeFi/Stake/NFT 合约交互频次
- 失败 / Gas 战士特征

派生人类可读标签(`DeFi 借贷玩家` / `ETH 质押者` / `NFT 收藏家` / `Gas 战士` / `链上元老` / `交易狂魔` ...)。算法独立、纯函数,可直接搬到任何 Ethereum 分析工具里。

### C. 已知合约 hash → 人类标签映射

`lib/etherscan.ts` 顶部维护了一份小型 dictionary:Uniswap V2/V3、Aave、Lido、OpenSea、USDC、Tornado Cash 等高频合约的 hash → 显示名。可继续扩展或共享。

### D. Anthropic 兼容协议直调样板

绕过 SDK 直接 `fetch` 调 `/v1/messages`,显式控制 `thinking`、`max_tokens`、`anthropic-version` header。对国内代理(MiMo、智谱等)兼容性最稳。代码在 `app/api/portrait/route.ts` 的 LLM 调用段。

### E. 卡片 React + html-to-image PNG 导出样板

`PortraitCard` 组件用 `forwardRef` 暴露 DOM ref,`html-to-image` 的 `toPng()` 一键导出。深色背景 + 渐变描边的视觉风格也可直接复用到其他"画像 / 名片 / 战绩"类生成器。

### F. Claude Code 全局 settings allowlist

`~/.claude/settings.json` 里那个 60+ 项的 Bash 白名单,任何重度用 Claude Code 的开发者都可以直接抄,避免 classifier 抽风时被 block。

## 六、如果重做一次

1. **直接走 OpenNext + Workers,跳过 Pages 这条死胡同。** 节省至少 3-4 次构建失败循环。
2. **本地先跑通 `npm run preview`(也就是 opennextjs-cloudflare build + preview)再 push**,而不是在云端调试。
3. **CF 项目创建时就同时配 Build vars 和 Worker runtime vars**,别等 build 通了才发现 runtime 拿不到。
4. **Wrangler / OpenNext 的"自动迁移"功能少用 / 别依赖。** 它会在云端构建机产生临时文件,但不回流到 repo,导致每次构建重做一遍。把 `wrangler.jsonc` / `open-next.config.ts` / 必要依赖都显式提交。
5. **截止日有时间硬约束的活动,先用最低保真版本 (`vercel deploy` 一行命令版本) 抢着把作品挂上去交付,再迭代。** 这次错过截止的根本原因是把"完美工程"放在了"按时交付"之前。
